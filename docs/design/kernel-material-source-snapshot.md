# Design: Material Intake And Source Snapshot

## Boundary

Material intake is a control-plane entrypoint. Source snapshot read is a kernel
tool primitive.

```text
Local path / upload / dropped file
        |
        v
Material Intake control surface
        |
        v
Source Snapshot owner
        |
        v
source_tree / source_read typed tools
        |
        v
bounded tool.result and provider context replay
```

The intake surface may accept a host path or upload stream because it runs with
operator authority. Model-visible tools never accept host paths. They accept
only source refs generated by the owner.

## Source Reference Model

The source owner projects:

```text
SourceSnapshotDescriptor:
  source_snapshot_ref
  display_label
  source_kind
  purpose
  entry_count
  total_uncompressed_bytes
  available_operations
  diagnostics

SourceFileDescriptor:
  source_file_ref
  path
  kind
  mime_type
  size_bytes
  text_readable
  available_operations
```

`available_operations` is a projection. `source_tree` and `source_read` must
validate the ref again, verify the underlying local file or uploaded object is
available, and re-check read budgets before producing output.

## Local Path Intake

The first locator shape is:

```text
material.intake({
  locator.kind = local_path
  locator.path = absolute host path
  purpose = source_analysis
  session_id? = optional session binding
})
```

Validation:

- `locator.kind` must be `local_path`.
- `locator.path` must be absolute and must identify a regular zip file.
- Missing or unreadable paths are refused with `resource_unavailable`.
- The host path is stored only in the source owner's private resolver state for
  live reads. It is absent from provider context, tool results, transcript,
  timeline, and model-visible descriptor metadata.

Local path snapshots are not durable body storage. If the original file is
deleted or changed later, source tools report the current live-read outcome.

## Zip Snapshot Parser

Zip parsing uses Go's standard `archive/zip` package. The parser normalizes zip
entry names with slash semantics and rejects unsafe names before they become
source descriptors:

- empty names;
- names containing `\`;
- absolute paths;
- Windows drive paths such as `C:...`;
- `..` segments or normalized paths that escape root;
- duplicate normalized file paths.

Budgets:

- maximum source file count;
- maximum per-file uncompressed bytes;
- maximum total uncompressed bytes;
- maximum tree result entries/bytes;
- maximum source read bytes.

Text detection is conservative: a NUL byte or invalid UTF-8 in the inspected
prefix makes an entry binary. Binary entries stay visible in tree output but
`source_read` returns a structured refusal instead of decoded noise.

## Typed Tools

Model-visible tools:

```text
source_tree({
  source_snapshot_ref: string,
  max_entries?: integer
})

source_read({
  source_file_ref: string,
  offset_bytes?: integer,
  limit_bytes?: integer
})
```

`source_tree` result includes snapshot ref, entries, truncation metadata, and
diagnostics. `source_read` result includes file ref, path, mime type, text,
offset, returned bytes, original bytes, truncation status, and next offset.

The tools share owner-side resolver and admission, but their schemas remain
separate. A universal `ref_read` would blur source, resource, job, artifact, and
debug refs, so it is rejected.

## Upload Intake

Upload intake is also a control surface. The initial implementation may use HTTP
multipart or an equivalent application command, but the contract is the same:

- upload bytes are written to a project-local object/file store;
- filename is a display label only;
- path traversal in filename is ignored or refused and never affects storage;
- oversized uploads are refused before snapshot parsing;
- zip uploads reuse the same source snapshot parser;
- non-zip binary uploads are not parsed as source snapshots.

Only refs, metadata, hashes, and bounded diagnostics are stored in facts. Full
upload bodies remain in object/file storage, not ledger or transcript.

## Provider And Projection Boundary

When a session has an admitted source snapshot, provider context may include a
bounded source-snapshot notice:

```text
kind = source_snapshot_context
text = "Source snapshot <ref> is available. Use source_tree and source_read..."
```

The notice contains no host path, storage ref, upload path, or raw payload id.
The model learns what operation to use, not where the file lives on the host.

Source tool results are ordinary tool results. Source intake facts are not
assistant messages, memory truth, connector delivery state, or approval
authority.

## Reference Alignment

- Reasonix `pastedFileRef` and `ResolveRefs` keep path detection and file
  resolution in the controller path. Genesis keeps that responsibility outside
  model-visible tools and emits source refs instead of prompt-spliced file
  blocks.
- Reasonix desktop dropped files distinguish workspace refs from attachments and
  bound preview serving with temporary tokens. Genesis uses opaque source refs
  and does not expose internal storage paths.
- Codex view-image and app-server filesystem tests route reads through
  environment/sandbox surfaces and fail when local FS support is unavailable.
  Genesis similarly refuses host-path reads unless the control surface has
  first admitted a snapshot.

## Rejected Directions

- Letting the model pass `D:\...zip` to `resource_read` or `source_read`.
- Treating `source_tree` and `source_read` as aliases of one generic `ref_read`.
- Persisting whole archive contents in the event ledger.
- Returning raw zip entry bytes for binary files.
- Encoding host paths or storage keys inside model-visible source refs.
- Implementing code search, OCR, vector indexing, or CodeGraph as part of this
  intake slice.
