# Kernel Issue Ledger

This file is the repo-owned ledger for active Genesis Kernel issues. Feishu Base is the collaboration inbox; this file is the durable project source for issues that still need code, verification, or user acceptance.

Retired issues must not remain here. Move accepted retirements to `docs/operations/kernel-retirement-log.md` as compact evidence: one sentence with the retirement conclusion plus the fixing commit evidence.

## Ledger Rules

- Keep only `new`, `open`, `in_progress`, or otherwise active issues in this file.
- Do not record application-specific feature work as kernel work unless it changes a kernel primitive.
- Do not add versioned HTTP route names as current contracts. HTTP is transport; current kernel endpoints are unversioned.
- `ready_for_acceptance` issues move to the retirement log as retirement candidates and leave this active ledger.
- Feishu/Base links may point to collaboration artifacts, but this repo must contain enough evidence to understand the current status without opening Feishu.
- Every active `KERNEL-*` issue must include a `Reference alignment` field that compares the issue to Codex, Reasonix, or an explicitly rejected drift risk.
- Reference alignment uses local reference checkouts only. Do not cite Genesis GitHub repositories, remote issues, releases, or pull requests as authority for this local kernel line unless the user explicitly asks for external publishing context.
- Before a non-trivial implementation slice starts, the related implementation plan or issue must include a Codex/Reasonix reference scan. The scan should identify inspected references, learned control-plane semantics, intentional differences, and remaining drift risks.
- Issues record the current gap between approved requirements/designs and the implementation. They must not carry raw requirements, design discussion, or the full production acceptance contract.
- Every active issue must cite an approved requirement and design unless it is an obvious bug or test gap. If an issue uses that exception, state the exception explicitly.
- Prefer `Gap`, `Next slice`, `Evidence`, and `Verification` over broad `Problem` or `Suggestion` text when adding new issues.
- Do not use issues as debug logs. Routine info, stream chunks, repeated status polling, and exploratory notes stay out of this ledger unless they identify a current implementation gap.
- When an issue removes a concept from the current kernel contract, long-term tests must assert the positive replacement behavior. Do not keep permanent tests whose only purpose is locking retired names, aliases, routes, or historical helper APIs out of the tree; use temporary scans or retirement-log evidence for cleanup windows, then fold the guard into the current owner contract.
- Development artifacts and historical local data are not compatibility obligations. Do not create or keep issues whose only purpose is migrating, cleaning, importing, or preserving old local generated state unless that state is part of the approved current kernel contract.
- Every implementation slice must finish with a drift check against the governing requirement, design, implementation plan, issue, and BDD feature. In-scope drift is fixed before commit. Out-of-scope drift is recorded here as an active issue with evidence and next slice before commit.
- Periodic governance review checks architecture, feature behavior, directory structure, and document lifetime together. Completed plans and stale documents should be deleted or condensed instead of spawning issues that only preserve old notes.

- Requirement: `docs/requirements/kernel-foundation-capabilities.md`.

- 标题: Provider reasoning output needs an explicit non-replay regression guard.
- 问题: Provider-native hidden reasoning such as DeepSeek `reasoning_content` is response-only data. Genesis currently appears to ignore it implicitly, but there is no executable guard proving it cannot enter durable transcript, provider context replay, provider command request JSON, OpenAI-compatible request messages, UI timeline, or audit replay.
- 建议: Add reference-inspired regression tests before future reasoning UI/streaming work expands the protocol. The tests should feed `reasoning_content` through the active OpenAI-compatible/provider-context path and prove only visible assistant content survives into model-visible replay.
- 证据: Reasonix has an explicit `TestBuildRequestDropsReasoningContent` guard; Genesis has provider/context projection tests but no test mentioning `reasoning_content`, `ReasoningContent`, or hidden reasoning non-replay.
- 验证: The future focused tests must fail if hidden reasoning text appears in `ProviderContextProjection`, provider command JSON, OpenAI-compatible request messages, session timeline, or audit replay.
- 优先级: P2.
- Gap: Genesis docs state that session transcript may store only product-approved reasoning summaries or notices and must not store provider raw payloads or hidden reasoning chains. The current OpenAI-compatible adapter ignores unknown provider response fields such as DeepSeek-style `reasoning_content`, but there is no explicit reference-inspired regression test proving response-only reasoning fields cannot enter `ModelResponse`, durable transcript, conversation history, provider command request, or a later OpenAI-compatible request. This is currently a test and guard gap, not evidence of an observed runtime leak.
- Next slice: Add focused tests around the active provider/context surfaces: an OpenAI-compatible response containing `message.reasoning_content` plus visible `content` must produce only the visible final text and normalized usage; a later same-session provider request must not contain `reasoning_content` or the hidden reasoning text; provider/context/audit/session projections must continue to omit hidden reasoning unless a future product-approved reasoning-summary event type is explicitly introduced with separate projection rules.
- Evidence: Local Reasonix has `internal/provider/openai/openai_test.go::TestBuildRequestDropsReasoningContent`, which proves assistant `ReasoningContent` is response-only and must not be echoed back to DeepSeek/OpenAI requests. Genesis currently has `TestOpenAICompatibleProviderCompletesAgainstCompatibleServer`, `TestOpenAICompatibleProviderNormalizesPromptCacheUsage`, `TestObservabilityProjectionsSeparateRawAuditAndProviderContext`, and `TestSubmitTurnProviderContextIncludesSameSessionHistory`, but no test mentioning `reasoning_content`, `ReasoningContent`, or a hidden reasoning non-replay case. `docs/field-reference.md` notes reasoning-token accounting is inspection-only and `docs/requirements/kernel-foundation-capabilities.md` says hidden reasoning chains do not enter transcript.
- Verification: Focused verification should include `go test ./internal/kernel -run "TestOpenAICompatible.*Reasoning|Test.*ProviderContext.*Reasoning|TestObservabilityProjectionsSeparateRawAuditAndProviderContext" -count=1`, followed by `go test ./internal/kernel -count=1` and `go test ./... -count=1`. The tests should fail if hidden reasoning text appears in `ProviderContextProjection`, provider command JSON, OpenAI-compatible request messages, session timeline, or audit replay.
- Reference alignment: Aligned with Reasonix's explicit DeepSeek/OpenAI `reasoning_content` non-replay guard and Codex's broader separation between model-visible context, UI/reasoning summaries, and raw protocol/event inspection. The active drift risk is future UI/streaming/reasoning-summary work accidentally treating provider-native hidden reasoning as conversation history or provider-visible context.

### KERNEL-PROVIDER-VISIBLE-FINAL-REPAIR-20260625 - P2 - Reasoning-only or empty final provider output should have a bounded visible-answer repair path

- Status: open.
- Area: Model Gateway / Provider Loop / Live LLM Usability.
- Requirement: `docs/requirements/kernel-foundation-capabilities.md`.
- Design: `docs/design/kernel-foundation-capabilities.md`.
- 标题: Reasoning-only or empty final provider output should have a bounded visible-answer repair path.
- 问题: The current OpenAI-compatible provider returns `provider returned no assistant content` when the provider response has no visible assistant `content`. That is safe, but it turns a model-visible repairable condition into a terminal provider failure. Reasoning-capable OpenAI-compatible providers can emit reasoning without visible final text; a user-facing turn should get a bounded repair step asking for a visible answer before the kernel fails the turn.
- 建议: Add a Model Gateway repair path and tests for empty/reasoning-only final responses. The first empty final should produce a model-visible repair request that asks for a visible final answer without replaying hidden reasoning. Repeated empty finals should stop after a small bounded count and record a structured turn failure.
- 证据: Reasonix `internal/agent/empty_final_test.go::TestRunRetriesReasoningOnlyFinalAnswer` retries after a reasoning-only answer and verifies the final assistant content becomes visible. `TestRunStopsAfterRepeatedEmptyFinalAnswers` proves the repair loop is bounded. Genesis has `openai_compatible.go` returning an error for empty assistant content and tests for provider failures/idempotent replay, but no visible-final repair test.
- 验证: Add focused tests where an OpenAI-compatible/provider fake first returns empty visible content and then returns visible text; the kernel should complete with the visible text and record repair evidence without exposing hidden reasoning. A repeated-empty case should fail with a structured reason after the bound, without tool execution, external effects, or unbounded provider calls.
- 优先级: P2.
- Gap: Genesis currently distinguishes provider failures from tool failures and records failed turns durably, but it lacks a bounded self-repair loop for provider responses that are syntactically valid yet lack visible assistant output. This is especially relevant before live LLM usage grows, because a reasoning-only result is not the same class as network/provider unavailability.
- Next slice: Specify the repair as Model Gateway behavior, not provider-adapter magic. The adapter should surface a typed empty-visible-output response or error class; the kernel loop should decide whether to issue a bounded visible-answer repair prompt, then record model context accounting and failure evidence without hiding repeated empty outputs.
- Evidence: `internal/kernel/openai_compatible.go` currently returns `errors.New("provider returned no assistant content")` for empty content. Current Genesis tests cover provider unavailable replay and live provider non-empty final smoke, but no test covers reasoning-only or empty-final repair. Reasonix has explicit reference tests for both retry and bounded stop.
- Verification: Future verification should include a focused `go test ./internal/kernel -run "Test.*VisibleFinal|Test.*EmptyFinal|Test.*ReasoningOnly" -count=1`, then `go test ./internal/kernel -count=1` and `go test ./... -count=1`.
- Reference alignment: Aligned with Reasonix's visible-answer repair loop for reasoning-only finals. The intentional Genesis difference should be that hidden reasoning remains non-replayed per `KERNEL-PROVIDER-REASONING-REPLAY-GUARD-20260625`; only the visible-answer repair instruction enters provider context.

- Reference alignment: Aligned with Codex background terminal list/terminate control surfaces and Reasonix's `ProgressFunc` plus session-scoped job manager. The active drift risk is turning live progress into provider-owned context, UI-owned truth, or a strong audit log instead of a kernel-owned durable fact with separate projections.
