# Kernel Issue Ledger

This file is the repo-owned ledger for active Genesis Kernel issues. Feishu Base is the collaboration inbox; this file is the durable project source for issues that still need code, verification, or user acceptance.

Retired issues must not remain here. Move accepted retirements to `docs/operations/kernel-retirement-log.md` as compact evidence: one sentence with the retirement conclusion plus the fixing commit evidence.

## Ledger Rules

- Keep only `new`, `open`, `in_progress`, or otherwise active issues in this file.
- Do not record application-specific feature work as kernel work unless it changes a kernel primitive.
- Do not add versioned HTTP route names as current contracts. HTTP is transport; current kernel endpoints are unversioned.
- `ready_for_acceptance` issues move to the retirement log as retirement candidates and leave this active ledger.
- Feishu/Base links may point to collaboration artifacts, but this repo must contain enough evidence to understand the current status without opening Feishu.
- Every active `KERNEL-*` issue must include a `Reference alignment` field that compares the issue to Codex, Reasonix, or an explicitly rejected drift risk.
- Reference alignment uses local reference checkouts only. Do not cite Genesis GitHub repositories, remote issues, releases, or pull requests as authority for this local kernel line unless the user explicitly asks for external publishing context.
- Before a non-trivial implementation slice starts, the related implementation plan or issue must include a Codex/Reasonix reference scan. The scan should identify inspected references, learned control-plane semantics, intentional differences, and remaining drift risks.
- Issues record the current gap between approved requirements/designs and the implementation. They must not carry raw requirements, design discussion, or the full production acceptance contract.
- Every active issue must cite an approved requirement and design unless it is an obvious bug or test gap. If an issue uses that exception, state the exception explicitly.
- Prefer `Gap`, `Next slice`, `Evidence`, and `Verification` over broad `Problem` or `Suggestion` text when adding new issues.
- Do not use issues as debug logs. Routine info, stream chunks, repeated status polling, and exploratory notes stay out of this ledger unless they identify a current implementation gap.
- When an issue removes a concept from the current kernel contract, long-term tests must assert the positive replacement behavior. Do not keep permanent tests whose only purpose is locking retired names, aliases, routes, or historical helper APIs out of the tree; use temporary scans or retirement-log evidence for cleanup windows, then fold the guard into the current owner contract.
- Development artifacts and historical local data are not compatibility obligations. Do not create or keep issues whose only purpose is migrating, cleaning, importing, or preserving old local generated state unless that state is part of the approved current kernel contract.
- Every implementation slice must finish with a drift check against the governing requirement, design, implementation plan, issue, and BDD feature. In-scope drift is fixed before commit. Out-of-scope drift is recorded here as an active issue with evidence and next slice before commit.
- Periodic governance review checks architecture, feature behavior, directory structure, and document lifetime together. Completed plans and stale documents should be deleted or condensed instead of spawning issues that only preserve old notes.

### KERNEL-TOOL-LOOP-PAUSE-RESUME-20260625 - P2 - Tool-loop round guard should pause resumably instead of failing production work

- Status: open.
- Area: Model Gateway / Interface Kernel / Work Recovery.
- Requirement: `docs/requirements/kernel-foundation-capabilities.md`; `docs/requirements/kernel-shell-and-job-control.md`.
- Design: `docs/design/kernel-foundation-capabilities.md`; `docs/design/kernel-shell-and-job-control.md`.
- 标题: Tool-loop round guard should pause resumably instead of failing production work.
- 问题: Genesis currently hard-limits model tool rounds with `maxModelToolRounds = 4`. When the provider still requests tools at that boundary, the kernel writes `turn.failed` with `tool_loop_limit_exceeded` and returns an error. This prevents infinite loops, but it also converts legitimate long-running exploration into a terminal failure instead of a resumable checkpoint. For an agent kernel, the guard should protect the system while preserving already committed tool evidence and giving the user/application a controlled continuation path.
- 建议: Define a bounded tool-loop pause/resume contract. When the configured tool-round budget is reached, the kernel should stop before executing the next tool batch, write a typed paused/interrupted/continuation-required fact with preserved provider-visible tool rounds, and expose a `continue` path through the normal `turn.submit`/session interface. Repeated continuations should remain budgeted and idempotent. The model should see prior committed tool calls/results on continuation, but hidden control ids and raw audit fields must stay out of provider context.
- 证据: `internal/kernel/model_tools.go` defines `const maxModelToolRounds = 4`. `internal/kernel/kernel.go` loops until that bound and, at the final round, appends `TurnError{Code: "tool_loop_limit_exceeded"}` before returning `model tool loop exceeded the maximum number of rounds`. Existing Genesis tests cover provider-visible tool rounds and idempotent retry, but the repository scan found no active issue or focused test for max tool-round continuation semantics. Local Reasonix treats positive `maxSteps` as a resumable guard: `internal/agent/agent.go::Run` returns a “paused after N tool-call rounds” message after preserving session state so the user can continue, and its coordinator tests assert the pause message. Codex similarly treats turns, interruptions, and resumes as first-class control surfaces rather than conflating all control stops with provider/tool failure.
- 验证: Add tests where a provider repeatedly emits valid tool calls beyond the configured round budget. Verify already committed tool results remain in ledger/provider projection, the boundary event is not `turn.failed` unless a true infrastructure error occurred, continuation can resume with the prior tool rounds visible to the provider, no duplicate side effects occur on retry, and hidden kernel ids remain absent from provider context. Include a repeated-continuation or malicious-loop case proving the guard remains bounded. Run focused tool-loop tests plus `go test ./internal/kernel -run "Test.*Tool.*Loop|Test.*ProviderContext.*ToolRound|TestHTTP.*Idempotency" -count=1`.
- 优先级: P2.
- Gap: The current round cap is a safety backstop, but its failure semantics are too coarse for production agent work. It lacks a durable paused/continuation state, an idempotent resume contract, and tests proving round-budget stops preserve committed facts without re-executing effects.
- Next slice: Write the requirement/design slice first: define the exact turn status, event type, retry/continue API shape, and provider-context projection rules. Then implement the smallest behavior test around a fake provider and a harmless registered tool before extending it to shell/job scenarios.
- Reference alignment: Aligned with Reasonix's `maxSteps` resumable pause semantics and Codex's control-plane separation for interruption/resume. Genesis should intentionally differ by expressing the pause as a kernel event/projection contract rather than a plain agent error string.

### KERNEL-TOOL-LOOP-STORM-GUARD-20260625 - P2 - Repeated tool failure and repeated write success need a model-visible loop guard

- Status: open.
- Area: Tool Runtime / Model Gateway / Safety Backstop.
- Requirement: `docs/requirements/kernel-foundation-capabilities.md`; `docs/requirements/kernel-shell-and-job-control.md`.
- Design: `docs/design/kernel-foundation-capabilities.md`; `docs/design/kernel-shell-and-job-control.md`.
- 标题: Repeated tool failure and repeated write success need a model-visible loop guard.
- 问题: Genesis has strong idempotency for explicit transport/tool retry keys, but it does not appear to detect model-level tool storms inside one user turn: the model can repeatedly emit semantically identical failing tool calls, or repeatedly emit the same successful write-like command with fresh kernel-generated tool-call event ids. Because each provider call receives a new kernel slot, ordinary idempotency does not stop a loop that burns the tool-round budget or repeats side effects.
- 建议: Add a kernel-owned storm-guard contract that observes consecutive tool outcomes within one turn. Repeated identical failure signatures should return a model-visible directive to change approach while preserving the original error; repeated successful write-like signatures should be blocked before another effect and returned as a non-executed guarded tool result. The guard must reset on real progress, partial success, a different batch shape, or a new turn. Shell-specific matching should start conservative: only clearly write-like commands or explicit non-shell write tools should enter repeat-success detection.
- 证据: `internal/kernel/model_tools.go` binds each model tool call to a fresh kernel `tool_call_event_id`; `shellInvokeModelToolResult` uses that idempotency key for the concrete shell operation. This protects replay of the same admitted operation, but not a model that reissues the same write command in later tool rounds with a fresh event id. Repository scans show tests for turn/shell/work idempotency and tool-loop ordering, but no test for repeated same-error failures or repeated same-success write calls inside a single turn. Local Reasonix has explicit storm guards in `internal/agent/storm_test.go` and `internal/agent/agent.go`: repeated same-error batches produce a `[loop guard]` model-visible result, and repeated identical write-like successes are blocked before another write.
- 验证: Add fake-provider tests that force three consecutive malformed/failing calls with different argument bytes but the same tool/error signature, and verify the later model-visible tool result contains a structured loop-guard directive rather than only the raw error. Add a second test where repeated same write-like shell intent would create duplicate files or append duplicate content; verify the repeat is blocked before a new effect and the user/session projection records guarded evidence. Add reset tests proving a successful different tool, partial success, or a new turn clears the guard. Run focused tool-loop/idempotency tests and then `go test ./internal/kernel -run "Test.*Tool.*Storm|Test.*Idempotency|Test.*Tool.*Loop" -count=1`.
- 优先级: P2.
- Gap: Current idempotency is retry-oriented, not behavior-oriented. It prevents duplicate effects for the same explicit key, but it does not catch a stuck model that generates fresh keys for the same failing or write-like action.
- Next slice: Specify the signature rules before implementation. For Phase A, guard repeated same-error tool batches and a narrow allowlist of write-like shell/write primitives; do not attempt broad semantic command equivalence for arbitrary shell text.
- Reference alignment: Aligned with Reasonix's storm breaker and repeat-success guard. Genesis should intentionally keep the guard inside Tool Runtime/Model Gateway outcome handling, not inside provider adapters or UI shells. Codex-style idempotent replay remains necessary but insufficient for model-level loop storms because provider-generated calls can be fresh while semantically repeating the same effect.

### KERNEL-CONTEXT-COMPACTION-STUCK-GUARD-20260625 - P2 - Auto compaction needs a stuck/cache-thrash guard for too-small windows

- Status: open.

- Design: `docs/design/kernel-foundation-capabilities.md`.

- 问题: Genesis currently has successful compaction, summarizer failure recording, failure backoff, provider-backed recent-tail accounting, and cache-stability evidence. It does not yet have an executable guard for the production case where compaction succeeds but does not reclaim enough effective context, causing every subsequent turn to compact again and repeatedly break prompt-cache locality. Failure backoff does not cover this case because the summarizer returns success.
- 建议: Add a compaction stuck guard owned by the kernel compaction runner. The first slice should be a requirement/design and behavior-test update: when a too-small context window or huge retained recent tail causes repeated consecutive successful compactions without effective headroom/cache recovery, the runner should pause/defer auto-compaction with typed evidence and a user/inspection notice. Manual compaction can remain available, but auto compaction should not thrash every turn.
- 证据: Genesis `internal/kernel/context_compaction.go` triggers auto compaction whenever provider-reported `InputTokens` crosses the limit, and `contextCompactionBackoff` only defers after `context.compaction.failed`. `TestAutoCompactionBacksOffAfterSummarizerFailure` covers failure retry/backoff, while `TestAutoCompactionRecordsUsageEconomicsAndCacheStability` records cache facts but does not assert that cache collapse or consecutive successful compactions pause future auto compaction. - 验证: Add focused tests that drive a long tool-heavy session with a context window too small for the retained tail. The tests should fail if auto compaction runs on many consecutive turns or keeps collapsing cache after successful compactions; they should pass only when the runner emits a `context.compaction.deferred` or equivalent paused evidence with a clear reason and leaves later provider context append-only until enough new evidence justifies retry. Companion tests should prove a healthy window still compacts when needed and does not pause.
- 优先级: P2.
- Gap: This is a production reliability and cost-control guard gap for the existing auto-compaction feature. It should not be solved in WebUI, provider adapters, or external shells; the kernel compaction runner owns the decision because it owns provider usage accounting, compaction state, and provider context projection.
- Next slice: Define the stuck criteria before implementing executor behavior. Candidate signals include consecutive successful auto compactions, compacted-region size versus retained-tail size, repeated compaction within the retry window, and cache-stability trend. Keep the first implementation conservative: prefer pausing auto compaction with typed evidence over trying to locally estimate token savings.
- Reference alignment: Aligned with Reasonix's stuck guard for too-small windows and cache-hit preservation tests. Codex also treats compaction as a lifecycle with analytics/status and replacement-history semantics rather than an unbounded loop. Genesis can keep its simpler provider-backed accounting model, but it should not intentionally drift into repeated successful compactions that provide no effective recovery.

### KERNEL-TOOL-PARALLEL-FAILURE-SEMANTICS-20260625 - P2 - Parallel pure-read batches need explicit per-call versus fatal failure semantics

- Status: open.
- Area: Tool Runtime / Tool Scheduling / Model-Visible Results / Recovery.
- Requirement: `docs/requirements/kernel-foundation-capabilities.md`.
- Design: `docs/design/kernel-foundation-capabilities.md`.
- 标题: Parallel pure-read batches need explicit per-call versus fatal failure semantics.
- 问题: Genesis now executes eligible `pure_read` batches concurrently and commits results in provider order, but the failure contract for a concurrent batch is not yet explicit. `executeToolBatchConcurrently` cancels the batch and returns an error on the first tool execution error, after which `handleToolExecutionError` records one turn-level failure and no sibling `tool.result` events are committed. Current `resource_read` turns ordinary malformed or unknown-resource cases into preflight repair results, so this gap is mainly about infrastructure failures and future non-shell `pure_read` tools. Without a written and tested boundary, future pure-read tools may accidentally turn a per-call read failure into a whole-turn failure, or conversely hide a fatal runtime failure as an ordinary tool result.
- 建议: Define a parallel-batch failure taxonomy before adding more parallel read tools. Non-fatal, model-actionable per-call failures should produce a `tool.result` for that call and keep sibling results ordered; fatal executor/runtime failures may fail the turn, but the contract must state whether already completed sibling pure-read observations are committed, discarded, or replayed. Add regression tests for at least: one pure-read succeeds while another returns a per-call failure; one pure-read hits `ErrToolInfrastructureFailed`; a runner returns duplicate/missing/unexpected results; and turn interruption cancels the batch without forging successful tool results. Keep `workspace_write`, `process_start`, `process_io`, `kernel_state_write`, and `external_side_effect` out of the concurrent runner until their idempotency/replay contracts are separately proven.
- 证据: Genesis `internal/kernel/tool_execution.go` runs concurrent pure-read batches through `executeToolBatchConcurrently`; any goroutine error cancels the context and returns a batch error, and `executeToolBatchesWithRunner` records a turn failure before committing any result from that batch. Existing tests prove planner fences, true pure-read concurrency, provider-order commits for successful out-of-order completions, serial prior-success commit before later serial error, and process-IO execution remaining serial by default. They do not cover concurrent partial failure semantics. Reasonix `internal/agent/agent.go` executes each call into a per-call `toolOutcome` and emits `ToolResult` in call order; its guards tests prove read-only parallelism while keeping writers and state-like read-only tools serial. Codex `codex-rs/core/src/tools/parallel.rs` converts non-fatal `FunctionCallError` into the failing call's output while keeping fatal join/runtime errors distinct.
- 验证: Add focused tests around `executeToolBatchesWithRunner` and at least one real `resource_read` path. The tests should fail if a model-actionable per-call read failure suppresses sibling successful results without a deliberate fatal reason, or if a fatal runner error is silently presented as a normal read result. Re-run `go test ./internal/kernel -run "TestExecuteToolBatches|TestPlanToolExecutionBatches|TestToolScheduling" -count=1`, then `go test ./internal/kernel -count=1` before retirement.
- 优先级: P2.
- Gap: This is a contract and test gap for the newly enabled concurrent execution path. The current implementation is conservative for `resource_read`, but the failure taxonomy must be locked before additional pure-read/resource primitives make parallel batches common production behavior.
- Next slice: Extend the Tool Scheduling requirement/design with per-call versus fatal parallel failure semantics, then add the regression tests before changing the runner. Do not solve this by exposing scheduling metadata to the model or by letting provider-native `parallel_tool_calls` decide failure behavior.
- Reference alignment: Aligned with Reasonix's per-call outcome model and Codex's per-call failure response for non-fatal tool errors while preserving Genesis's stricter effect-class scheduler. The active drift risk is collapsing all parallel-batch failures into a single `turn.failed` or, in the opposite direction, treating executor/runtime failures as ordinary model-repairable read results.
